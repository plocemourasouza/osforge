# Config Critique — LLM-Powered Linter for OSForge Customizations

> Expert reviewer that evaluates the user's customized configurations before
> they become source of truth. 4 axes. Constructive. Concise.

## When to use

| Trigger | What it critiques |
|---|---|
| User created a new SKILL.md | Frontmatter, description triggers, conflicts with existing skills |
| User wrote a `.mdc` rule | Globs, `alwaysApply`, conflicts with base rules |
| User customized hooks in `settings.json` | Pattern matching, false positives, edge cases |
| User added an agent `.md` | Reality Check + QC Loop present, persona conflicts with others |
| User overrode a CLAUDE.md section | Conflict with SKILLS.md, with rules |
| Before a PR to OSForge upstream | Quality gate for PR review |

## When NOT to use

- Configs auto-generated by `deploy.sh` (already validated)
- Trivial edits (fixing a typo, changing 1 line)
- Immutable built-in skills of OSForge core

## The 4 axes

### 1. Clarity
> Is the rule/skill unambiguous? Could any LLM misinterpret it?

Look for:
- Vague descriptions ("useful for design things")
- Ambiguous triggers (words that appear in any prompt)
- Passive verbs ("can be used") instead of imperatives ("Use when")
- References without context ("use the default pattern" — which one?)
- Frontmatter fields that are empty or inconsistent

**Good:**
```yaml
description: >
  Use when: "create premium landing page", "Awwwards style",
  "magnetic hero". Loads 3-dial system (VARIANCE/MOTION/DENSITY).
```

**Bad:**
```yaml
description: Good for nice design
```

### 2. Completeness
> Are there uncovered gaps or edge cases?

Look for:
- Skills without "When NOT to use"
- Rules without a fallback when the pattern doesn't match
- Hooks without error handling
- Agents without "What this agent does NOT do"
- Missing negative examples
- Triggers that cover only Portuguese OR only English (in OSForge they should cover both)

**Test question:** *If I hand this SKILL.md to a fresh LLM, can it decide ON ITS OWN when to activate?*

### 3. Conflicts
> Does it conflict with existing rules/skills?

Look for:
- Trigger overlap (2 skills competing for the same prompt)
- Rules with overlapping globs
- Agents with duplicate personas
- Hooks that block a command another hook allows
- CLAUDE.md section overrides that invalidate `alwaysApply` rules

**Cross-reference against the 122 existing skills:**
```bash
# Grep similar triggers
grep -r "your-new-trigger" skills/*/SKILL.md
```

### 4. Actionability
> Specific enough to act on?

Look for:
- Rules without concrete instructions ("be careful with X")
- Skills without clear outputs (doesn't say what to deliver)
- Agents without "done" criteria
- Triggers without a decision tree ("if A, then B; if C, then D")
- Pseudo-code without real code

**Good:**
```yaml
description: >
  Use on: TypeScript strict violations. ACTION: run `tsc --noEmit`,
  parse output, list errors prioritized by severity, propose a fix
  for each.
```

**Bad:**
```yaml
description: Takes care of TypeScript quality in the project
```

## Critique protocol

### Step 1: Identify the config type

```python
config_type = detect_config_type(file_path)
# possíveis: skill, rule, hook, agent, claude_md_override
```

### Step 2: Load baseline for comparison

```python
baseline = {
    "skill": load_all_skills_index("claude-code/SKILLS.md"),
    "rule": load_all_rules("rules/*.mdc"),
    "hook": load_hooks_json("hooks/hooks-claude-code.json"),
    "agent": load_all_agents("agents/*.md"),
}[config_type]
```

### Step 3: Apply the 4 axes with structured output

```yaml
critique:
  file: skills/my-new-skill/SKILL.md
  overall_grade: B+   # A / B+ / B / C+ / C / D / F
  axes:
    clarity:
      score: 7/10
      issues:
        - line: 4
          severity: high
          message: "Description 'useful for tasks' is vague. Replace with specific triggers."
          suggestion: "Use when: '...', '...', '...'"
    completeness:
      score: 8/10
      issues:
        - severity: medium
          message: "Missing 'When NOT to use' section"
          suggestion: "Add a section between 'When to use' and 'Protocol'"
    conflicts:
      score: 6/10
      issues:
        - severity: high
          message: "Trigger 'premium design' conflicts with skill `taste-design-dials` (line 3)"
          suggestion: "Differentiate: this skill is for X, taste-design-dials is for Y. Adjust both descriptions."
    actionability:
      score: 9/10
      issues: []
  recommendation: "Iterate Clarity + Conflicts before merging. Completeness is nice-to-have."
```

### Step 4: Final recommendation

| Grade | Action |
|---|---|
| **A / A-** | Approve |
| **B+ / B** | Approve with optional suggestions |
| **B- / C+** | Requires iteration — list mandatory fixes |
| **C / C-** | Requires rewrite — structural issues |
| **D / F** | Block — serious conflicts or broken config |

## Golden rule

Comment ONLY on rules that can be improved. Do not invent issues to show work. Concrete example — config with 3 axes OK and 1 real issue:

**ACCEPTABLE output:**
```
Grade: B+. Clarity, Completeness and Actionability: OK.
Conflicts (line 3): trigger "premium design" collides with `taste-design-dials`.
Fix: narrow the description to "landing pages with motion design".
```

**REJECTED output:**
```
Grade: B+. Let's analyze each axis in depth...
[3 paragraphs praising Clarity] [2 paragraphs speculating that
Completeness "could perhaps be more robust"] [a section on "general
considerations about skill best practices"] ...and finally, there is a trigger conflict.
```

The first delivers the same fix in 4 lines; the second buries the single real issue under a rant and invented issues.

## Anti-patterns

- ❌ Generic critique ("could be clearer") — no location or fix
- ❌ Sweeping critique when only 1 issue exists — a rant
- ❌ Accepting broken config because "it's close" — a gate is a gate
- ❌ Checking syntax only — semantics > syntax
- ❌ Ignoring conflicts with existing configs (this is the most important axis)

## Meta-verification (critique of the critique itself)

Before delivering:
- [ ] Each issue has a location (line, file, section)
- [ ] Each issue has a severity (low/medium/high/critical)
- [ ] Each issue has a concrete suggestion
- [ ] Cross-reference with the baseline was done
- [ ] Output is parseable YAML/markdown, not loose prose

## Workflow integration

### Manual mode (interactive)
```
User: "review this skill I wrote: skills/my-thing/SKILL.md"
Claude: [loads config-critique skill] [applies the 4 axes] [returns structured output]
```

### Automatic mode (pre-commit hook)
Add to `.husky/pre-commit` or to the OSForge hook:
```bash
# If files in skills/, rules/, agents/ were modified:
modified=$(git diff --cached --name-only | grep -E '^(skills|rules|agents)/')
for file in $modified; do
  osforge-db critique "$file" || exit 1
done
```

### PR mode (CI gate)
A GitHub Action that runs config-critique on PRs touching skills/rules/agents.
